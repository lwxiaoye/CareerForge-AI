# AI 面试官回答形式优化方案

审计日期：2026-06-12

目标：让 AI 面试官的交互更像“会说人话、能解释动作、有实时反馈”的简历助手，但仍保持面试官的严格角色。

## 1. AI 面试官和简历助手聊天形式差异

### 1.1 简历助手当前形态

简历助手走 `AgentChatView + chatRuntimeStore + RunManager + SSE`：

- 会话惰性创建。
- 消息是流式 delta。
- 工具调用有 activity 胶囊。
- 可以展示“正在读简历、正在修改、已完成”。
- 绑定工作简历后，操作对象清晰。
- 修改后可以跳转简历中心、撤销。

关键文件：

- `frontend/src/student/AgentChatView.tsx`
- `frontend/src/student/chatRuntimeStore.ts`
- `backend/app/student/run_manager.py`
- `backend/app/student/agent_runtime.py`

### 1.2 AI 面试官当前形态

AI 面试官走独立页面和独立 API：

- `POST /student/interviews` 创建面试，一次性返回第一题。
- `POST /student/interviews/{id}/turns` 提交回答，一次性返回下一题。
- 没有 SSE。
- 没有工具 activity 胶囊。
- 前端用 `reportProgress` 和 `interviewProgress` 模拟状态。
- 用户回答已做乐观展示，但 AI 下一题不是流式。

关键文件：

- `frontend/src/student/AIInterviewerPage.tsx`
- `backend/app/interview/router_student.py`
- `backend/app/interview/service.py`

## 2. 可以模仿简历助手的地方

### 2.1 回答前先展示动作状态

简历助手的优秀体验是“先告诉用户正在做什么”。面试官也应该如此，但不需要暴露内部 Harness 细节。

建议状态文案：

- 已收到你的回答
- 正在匹配岗位 JD 和简历证据
- 正在检查回答是否空泛
- 正在组织下一轮追问
- 正在保存面试记录

技术上可以先做前端状态流，后续再做后端 SSE。

### 2.2 每轮回答后给短反馈，再追问

当前 AI 面试官主要返回下一问，但可以优化成“反馈 + 追问”的固定结构：

```text
你这轮提到了 Redis 缓存，这是有效线索。
但还缺两个关键信息：你的个人动作、优化前后的数据。

下一问：
请围绕这个接口优化，补充你具体做了哪一步，以及 P95/错误率/吞吐量有没有变化。
```

这和简历助手“做前预告、做后总结”的节奏一致。

### 2.3 每轮展示“为什么问”

后端已经有字段：

- `question_reason`
- `capability_tags`
- `top_sources`

序列化已经返回：

- `backend/app/interview/service.py:151-169`

但前端没有充分展示。建议在 AI 问题气泡下方展示：

- 考察点：项目证据 / 技术细节 / 岗位匹配
- 追问原因：上一轮缺少量化指标
- 题库来源：Redis 面试题库 / 缓存一致性

### 2.4 报告生成过程要像“生成中”，不是“查询中”

当前按钮叫“结束并生成报告”，但实现调用：

- `GET /api/v1/student/interviews/{id}/report`
- `frontend/src/student/AIInterviewerPage.tsx:589`

如果报告未生成，后端返回“报告不存在”：

- `backend/app/interview/service.py:1492-1503`

应改为：

- 用户点击结束并生成报告。
- 前端调用 `POST /api/v1/student/interviews/{id}/finish`。
- 后端真正执行 `generate_report(...)`。
- 生成后展示报告。

## 3. 面试官回答模板建议

### 3.1 开场问题模板

目标：不泛泛自我介绍，直接围绕简历和 JD。

```text
我已经读取了你选择的简历和目标岗位 JD。今天按「一面」来，我会先验证简历真实性和岗位基础匹配。

第一问：
请选一个最能证明你适合「{target_role}」的项目，按背景、你的职责、关键方案、量化结果说明。
```

### 3.2 追问模板

```text
你这轮回答有一个有效点：{positive_point}。
风险是：{risk_point}。

下一问：
{next_question}
```

要求：

- 不给标准答案。
- 不替用户编经历。
- 不说“你已经通过面试”。
- 每次只问一个主问题。

### 3.3 空泛回答模板

```text
这轮回答偏空泛。你用了「负责、参与、熟悉、优化」这类词，但没有给出个人动作和结果数据。

下一问：
请只围绕一个细节补充：你当时具体改了哪一处设计或代码，优化前后有什么可量化变化？
```

### 3.4 错误回答模板

```text
这个回答里有明显风险：{risk_point}。
我先不给标准答案，你有一次修正机会。

下一问：
请重新解释 {topic} 的关键机制，并说明它在你的项目中为什么适用。
```

### 3.5 报告摘要模板

```text
本轮综合分：{score}

最需要补强的是「{weakest_dim}」。主要原因不是不会说，而是证据不够硬：缺少个人动作、量化指标和技术取舍。

下一轮训练建议：
1. 先重写最低分问题的回答。
2. 准备一个带数据的项目案例。
3. 再练一场二面，重点追问项目深度。
```

## 4. 产品形态建议

### 4.1 不要把面试官做成简历助手的同款自由聊天

面试官必须是结构化流程：

- 目标岗位
- JD
- 简历
- 面试类型
- 阶段
- 每轮回答
- 每轮评分
- 报告

可以模仿简历助手的“表达节奏”和“状态反馈”，不要模仿简历助手的自由工具调用。

### 4.2 UI 展示顺序

每轮 AI 气泡建议分四层：

1. 简短反馈
2. 下一题
3. 考察点 tag
4. 展开后看评分原因/题库来源

每轮用户气泡：

1. 用户回答原文
2. 提交中/已保存/失败状态

报告区：

1. 总分和最低维度
2. 维度雷达或条形
3. 优势
4. 风险
5. 训练计划
6. 下一场预设

## 5. 和牛客流程的参考对齐

参考牛客企业版公开资料：

- 牛客 AI 面试强调“解析岗位 JD，自动生成针对性问题”，避免无关闲聊。
- 强调动态追问，根据候选人回答实时调整方向。
- 强调 360° 智能评估报告，标注亮点和风险。
- 牛客产品页还强调 AI 简历筛选的人岗匹配和硬性条件评估。

来源：

- https://hr.nowcoder.com/article/9014
- https://hr.nowcoder.com/

CareerForge 当前已部分具备：

- JD 必填。
- 读取简历快照。
- RAG 检索题库。
- 动态追问。
- 结构化报告。

缺口：

- 前端没有指定哪份在线简历。
- 报告按钮没有触发生成。
- 问题原因/题库来源展示不充分。
- 真正语音/流式还未完成。
