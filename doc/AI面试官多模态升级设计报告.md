# AI 面试官多模态升级设计报告

**项目**：CareerForge-AI
**日期**：2026-06-11
**范围**：AI 面试官升级方案——文字面试 + 豆包式「数字人视频通话面试」（语音、数字人、视频通话三者整合为一个功能）

---

## 一、产品形态：只有两种，不是四种

| 形态 | 体验 | 技术本质 |
|------|------|----------|
| **文字面试** | 现有对话界面，面试官文字提问、学生打字回答 | 现有 Agentic Loop，补面试编排即可 |
| **通话面试** | 像豆包视频通话：学生点「开始面试」即接通一路电话，对面是**数字人面试官**，全双工语音、随时打断；学生可选开摄像头 | 端到端实时语音模型 + RTC 传输 + 前端数字人渲染 |

关键认知：**语音面试、数字人、视频通话不是三个功能，是一个功能的三个层**——

```
┌─────────── 通话面试（一个功能）───────────┐
│  画面层：数字人渲染（面试官的"摄像头画面"）  │ ← 前端组件，吃音频流驱动嘴型
│  音频层：全双工实时语音（豆包式打电话）      │ ← 实时模型 + RTC，功能的核心
│  会话层：面试编排 / 简历上下文 / 会后报告    │ ← 复用现有 Agentic Loop 资产
└──────────────────────────────────────────┘
```

「纯语音面试」只是通话面试把数字人画面收起的退化形态（一个 UI 开关），不需要单独立项开发。

**核心结论**：
1. 豆包式视频通话面试**可以实现**，国内有现成商用方案（火山引擎对话式 AI / 智谱 GLM-Realtime），不需要自研实时模型或自建媒体服务器。
2. 数字人推荐**前端实时渲染**（Live2D/VRM），用通话下行音轨驱动嘴型，零额外延迟、零边际成本——这正是它能和实时语音天然整合的原因。
3. 通话期间会绕开现有 Agent Loop，需要「会前注入上下文、会后回灌记录」的配套设计（第四节），让面试记录和报告回到现有体系。
4. **RAG 不是必需的**；做「公司真题库面试」时以一个检索工具轻量加入（第六节）。

## 三、第一步：文字面试补「面试编排」（所有形态的根基）

文字对话能力已经有了，缺的是**结构化的面试流程**。而且通话面试的 system prompt 直接复用这套编排，所以必须先做：

1. **面试配置入口**：开始前让学生选岗位方向、面试类型（HR面/技术面/综合面）、时长/轮次。前端在 `AgentChatView`（`agentType="interviewer"`）空状态处加配置卡片。
2. **面试状态机**（prompt 层实现即可）：开场 → 自我介绍 → 简历深挖（调 `read_resume`/`query_student_profile`）→ 岗位题 → 反问 → 结束。在 `INTERVIEWER_SYSTEM_PROMPT` 明确「一次只问一个问题」「根据上一答案追问」「时间到进入总结」。
3. **面试报告**：结束后生成结构化评价（STAR 完整度、表达逻辑、岗位匹配度），复用 `export_resume_pdf` 的 reportlab 管线导出 PDF。

零新技术栈，纯 prompt 工程 + 少量前端，约 1 周。

## 四、通话面试：豆包式数字人视频通话

### 4.1 豆包体验的技术拆解

豆包视频通话 = ①全双工语音（<1s 响应、随时打断）+ ②摄像头视频理解 + ③通话型 UI。背后是**端到端实时多模态模型 + RTC 传输**，不是「ASR→LLM→TTS」三段拼接——三段拼接延迟 1.5–3s 且打断体验差，做不出"打电话"的感觉。所以通话面试的传输层用 WebRTC/WebSocket（取代 HTTP/SSE），模型用 realtime 形态。

### 4.2 实时模型选型（国内可用）

| 方案 | 说明 | 视频理解 | 备注 |
|------|------|----------|------|
| **火山引擎「对话式 AI」**（RTC + 豆包端到端实时语音模型） | 豆包同款技术栈；Web/移动端 RTC SDK，内置回声消除（AEC）、VAD、打断（barge-in） | 配合视觉模型可实现 | **首选**：体验最接近豆包，文档/demo 完善 |
| **智谱 GLM-Realtime** | WebSocket 实时 API，原生支持**摄像头视频流输入** + 语音 + function calling | ✅ 原生 | 做「AI 看得见学生」的二期首选 |
| 阿里 Qwen-Omni Realtime（DashScope） | 实时音频对话 API，Omni 系列支持视觉 | 部分 | 备选 |
| OpenAI gpt-realtime | 效果标杆 | ✅ | 国内网络/合规不便，不推荐主线 |

推荐路径：**一期用火山对话式 AI 做纯语音通话（模型只听不看），二期要"AI 能看到学生"再评估 GLM-Realtime**。两者都支持 function calling，可注册只读工具。

### 4.3 数字人画面层：前端渲染，不是视频生成

| | 路线 ①：服务端视频生成（MuseTalk / HeyGen 类 SaaS / 腾讯智影） | 路线 ②：前端实时渲染 + 音频驱动嘴型（**推荐**） |
|---|---|---|
| 原理 | 每句话生成口播视频推流给前端 | 浏览器渲染 2D/3D 形象，本地分析下行音轨实时驱动嘴型 |
| 延迟 | 每句额外 2–10s，**和实时语音的低延迟直接矛盾** | ≈0，与音频严格同步 |
| 成本 | GPU/按秒计费，贵 | 纯前端，零边际成本 |
| 效果 | 真人照片级 | 卡通/动漫风（校园产品更友好、无恐怖谷） |

路线 ② 是数字人能与实时语音**整合**的前提——它只消费音频流，不在链路上增加任何延迟。选型按投入从低到高：

1. **Live2D**（pixi-live2d-display）：社区有大量免费模型，嘴型用 `MouthOpenY` 参数绑定音量包络，一周可出 demo。**建议起步**。
2. **VRM 3D**（@pixiv/three-vrm + three.js）：VRoid Hub 免费模型，支持 blendshape 表情（微笑/点头/思考），嘴型可升级共振峰分析，更精细。
3. SaaS 真人数字人（硅基 Duix / 腾讯智影直播数字人）：真人级但成本高、依赖重，远期再说。

### 4.4 整体接入架构

```
┌──────────────── 浏览器（通话界面）────────────────┐
│ getUserMedia：麦克风（必选）+ 摄像头（可选，一期仅   │
│               本地画中画自检，不上传）              │
│                                                   │
│ RTC SDK ←──── 上行音频 ────→ 实时模型（火山/智谱）  │
│    │                              │               │
│    │ 下行音轨（面试官语音）         │ 转写/事件回调   │
│    ├──→ 扬声器播放                 │               │
│    └──→ AnalyserNode ──→ AvatarRenderer（数字人）  │
│              音量/频谱      ├ 嘴型：音量包络→MouthOpenY│
│                            ├ 表情：avatar.emotion 事件│
│                            └ 待机动画：眨眼/摆动      │
└────────────┬──────────────────────────────────────┘
             │ 会前：POST /student/realtime/sessions
             ▼
┌──────────────── FastAPI 后端 ─────────────────────┐
│ ① 会前：读档案+简历 → 拼面试官 system prompt（复用   │
│    第三节编排）→ 向供应商申请临时 token → 返回前端    │
│ ② 会中：接收转写回调，把每轮问答写入                  │
│    StudentAgentMessage（复用现有消息表）             │
│ ③ 会后：完整 transcript 喂给现有 run_agent_loop()    │
│    生成面试评估报告 PDF（复用 reportlab 管线）        │
└────────────────────────────────────────────────────┘
```

四个关键设计点：

1. **音视频流不过我们的后端**。浏览器直连供应商 RTC（后端只签发临时 token），FastAPI 不碰音频帧，否则要自己做实时转发，延迟和复杂度都失控。
2. **Harness 被绕开的补偿**。通话期间模型直接对话，不走 Agent Loop，补偿三招：
   - **会前注入**：把 `query_student_profile` + `read_resume` 结果在建会话时拼进 system prompt（面试上下文就这两样，体积可控），通话中无需调工具；
   - **会中兜底**：实时 API 的 function calling 注册只读工具，模型确实需要时仍可查；
   - **会后回灌**：转写写回 `StudentAgentMessage`，历史、报告、审计全部回到现有体系。
3. **数字人与语音的整合点就一处**：下行音轨 → `AnalyserNode` → `AvatarRenderer`。数字人组件对"音频从哪来"无感知，纯语音模式 = 同一界面隐藏画布。
4. **表情控制**：让实时模型在回复文本里输出 `<emotion:smile>` 类轻量标记（system prompt 约定），前端从转写事件中剥离并驱动数字人表情；没有标记时走待机动画，不影响主流程。

### 4.5 改动清单

后端（新增约 2 个文件 + 1 条迁移）：
- `backend/app/student/realtime_router.py`：`POST /student/realtime/sessions`（建会话、签 token）、转写回调接收端点。
- `core/realtime_client.py`：参照 `llm_client.py` 封装供应商「会话创建层」（多供应商抽象只做到这层，SDK 层换供应商需改前端）。
- 迁移：session 加 `interview_mode`。

前端（新增约 3 个组件）：
- `CallInterviewView.tsx`：全屏通话界面（数字人画布、通话计时、静音/挂断、可选自检画中画）。
- `AvatarRenderer.tsx`：Live2D 封装，输入 AnalyserNode + emotion 事件。
- RTC SDK 接入层（火山 Web SDK）。

资源：1–2 个可商用 Live2D 模型（约 5–20MB，放 `frontend/public/avatars/`）。

### 4.6 现实约束（排期里要讲清楚）

- **成本**：实时模型按音频分钟/token 计费，比文字贵一个量级。建议管理端给通话面试设每人时长配额（如每天 2 场 × 20 分钟）。
- **网络**：RTC 对弱网敏感。前端做质量检测，弱网时提示「切到文字面试」。
- **降级方案（可选）**：若想要一个不依赖 RTC 的低配语音版（按住说话 → ASR → 现有 Agent Loop → TTS 播放，延迟 1.5–3s），可以用 HTTP/SSE 实现且 Harness 全链路保留——但它是备胎，不是主线，建议只有在 RTC 方案落地受阻时才投入。
- **摄像头双向（二期）**：模型看学生（GLM-Realtime）可做「眼神/语速」反馈，是差异化卖点，但涉及人脸数据，必须显式授权 + 明示不存储。一期只做单向（学生看数字人，模型只听）。

### 4.7 工作量评估

依托供应商 RTC SDK：通话面试 MVP（实时语音 + Live2D 数字人 + 会前注入 + 会后报告）约 **4–5 周**；二期摄像头视频理解 +1–2 周。前提是第三节的面试编排已打磨好。

## 六、需不需要 RAG？

### 不需要的部分（占 80%）
- **简历深挖**：已有 `read_resume` / `query_student_profile` + `SessionEvidencePool`，本质是结构化精准检索，比 RAG 更可靠（无召回失败）；通话模式下这些内容会前注入 prompt。
- **通用面试对话**：大模型自身知识足够撑 HR 面和常见行为题。
- 实时语音、数字人管线本身与 RAG 完全无关。

### 有价值的部分（可选增强）
**面试题库 / 面经知识库**：想让面试官「按某公司某岗位真实面经提问、按考点评分」，题库超过几百题塞不进 prompt，此时 RAG 合适：
- **存储**：万级以内题量用嵌入式向量库——**sqlite-vec** 或 **Chroma**，零运维，符合「本地 SQLite 开发」习惯，不必上向量数据库集群。
- **Embedding**：模型广场新增 `capability="embedding"`，选 BGE-M3 / 阿里 text-embedding-v3。
- **接入方式**：文字面试做成工具 `search_question_bank(岗位, 主题, 难度)` 由 Agent Loop 自主调用；通话面试在会前按面试配置预检索 5–10 题塞进 system prompt。
- **数据来源**：管理端加「题库管理」页，按 `tenant_id` 隔离。

**结论**：核心功能不需要 RAG；做「公司真题库面试」时以检索工具 + 嵌入式向量库轻量加入。

## 七、实施顺序建议

1. **面试编排**（约 1 周）：prompt 工程 + 配置 UI + 面试报告。文字面试直接受益，通话面试的 prompt 也在这里打磨。
2. **通话面试 MVP**（约 4–5 周）：
   - 第 1–2 周：火山对话式 AI 跑通（建会话/token/通话/转写回灌），纯音频无数字人；
   - 第 3 周：Live2D 数字人接入下行音轨（嘴型 + 待机动画），通话 UI 成型；
   - 第 4–5 周：表情标记、时长配额、弱网提示、会后报告打通。
3. **二期增强**：摄像头视频理解（GLM-Realtime，1–2 周）、VRM 3D 形象升级、RAG 题库（1–2 周）。

文字面试与通话面试共享面试编排、消息表、报告管线；通话面试通过「会前注入、会后回灌」与现有 Agentic Loop 闭环，不产生架构分叉。
