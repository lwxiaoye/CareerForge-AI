## 平台协作（主智能体必读）

- **底稿来源（不强制已有简历，支持从 0 订制）**：
  1. 先调用 `read_resume`；若学生已有简历，用它作底稿。
  2. **若没有简历，也要能从 0 帮学生订制**：用学生本轮消息里已给的背景，或主动提问收集其真实背景（工作/实习经历、教育、技能、项目、成果），据此构建简历。**绝不能因为没有简历就拒绝服务**。
- **必须有目标 JD**：本 Skill 的核心是"贴着某个具体 JD 定制"。若学生没给 JD/岗位要求，先请其补充，**绝不编造 JD**。
- **真实性优先**：只使用学生真实提供的信息做重组、强调和措辞优化；信息缺失就**问学生**，**绝不编造经历、公司、联系方式或量化数字**。
- **可下载版**：完成后若学生要 PDF，调用 `export_resume_pdf`（按下方固定模板、一页排版），把 download_url 以 Markdown 链接给学生。

# 订制简历生成器

> 改编自 Claude Code 社区 Skill `tailored-resume-generator`，已适配本平台（中文、真实性约束、固定 PDF 模板、一页）。

## 何时使用本 Skill

- 投递某个**具体岗位**，需要一份"贴着 JD 改"的简历。
- 跨行业/转岗，需要突出可迁移经历。
- 想让简历更容易通过 ATS（关键词筛选系统）。

> 区分：只想笼统润色 → 用「简历优化」；只想看匹配度评分诊断 → 用「岗位匹配」；要**针对某岗位定制生成一份可投递简历** → 才用本 Skill。

## 工作流程

### 1. 收集信息
- 目标 JD（必须）+ 公司名 / 岗位名。
- 学生背景：优先 `read_resume`；**若没有简历，就用学生消息里已给的背景，或主动提问补齐**——一次问 3–6 个关键问题即可（工作/实习经历、教育、技能、项目、成果、证书），不要一次问太多。
- 若学生只给了 JD 没给背景：先简短列出你需要的几项信息请其补充，拿到后**从 0 生成**一份贴合该 JD 的简历。

### 2. 解析 JD（分优先级）
- **必须项**：硬性要求（年限、必备技能、学历）——缺失是硬伤。
- **重要项**：强烈期望的技能/方法论。
- **加分项**：nice-to-have。
- 同时提取：硬技能关键词、职责动词、行业领域、可用于 ATS 的高频词。

### 3. 把经历映射到要求
- 每条 JD 要求 → 在简历里找对应或可迁移的经历。
- 标出无法满足的差距（弱化展示或如实说明，不编造填补）。
- 找出能让学生脱颖而出的独特优势。

### 4. 组织定制后的简历
- **专业摘要（3–4 行）**：以目标角色的年限/方向开头，嵌入 JD 中最关键的 3–4 个技能，点明行业经验与独特价值。
- **技能栈**：按 JD 关注点分类，必备技术放最前，**用 JD 里的原词**，只列能用经历佐证的技能。
- **工作 / 项目经历**：强动词开头，按「动作 + 技术 + 过程 + 结果」写；把与 JD 最相关的经历和要点**提到前面**；自然嵌入 JD 关键词；**只对真实经历做重组与强调**。
- **教育背景**：相关学历/课程/证书。

### 5. ATS 优化
- 用标准板块名（专业技能 / 工作经验 / 项目经历 / 教育背景）。
- 自然嵌入 JD 原词；必要时缩写 + 全称都写（如 "Redis（缓存中间件）"）。
- 避免花哨表格/图形/页眉页脚（本平台 PDF 模板已是 ATS 友好的简洁版式）。

### 6. 策略建议（生成简历后附上）
- **优势**：最有竞争力的点，可在求职信/面试展开。
- **差距 + 弥补**：未满足的要求，给可执行的补强方向。
- **面试谈点**：基于简历值得准备的故事。
- **求职信开头**：给 2–3 句可用的开场。

## 真实性铁律（平台强约束）

1. 绝不编造经历、公司、实习、联系方式。
2. 量化只能用学生真实提供的数据；没有数据就用"提升效率 / 降低成本 / 优化响应"等定性表达，**严禁瞎编百分比、用户数、营收**。
3. 不夸大可迁移能力，不把"了解"写成"精通"，不把"参与"写成"主导"。
4. 不写第一人称（我/我的），不放年龄/婚姻/照片等无关信息。

## 篇幅

- 经验 < 10 年 → **严格一页**；只保留与目标岗位最相关的 2–4 段经历，其余精简或删除。

## 不同人群侧重

- **应届生**：教育背景靠前，突出项目、实习、竞赛、学生组织领导经历。
- **转岗者**：突出可迁移技能，在摘要里讲清转型逻辑与动机。
- **技术岗**：技能栈板块靠前，附 GitHub / 作品集链接，可提方法论（Agile/Scrum）。

## 生成 PDF 的固定模板（调用 export_resume_pdf 时务必严格按此 Markdown 格式）

渲染器会把下面的结构排成专业版式：头部分栏、灰色板块标题条、日期自动右对齐、整体一页。**必须严格遵守，否则排版会乱。**

```
# 姓名
职位头衔
求职状态 | 邮箱 | 电话 | 城市 | 个人主页或GitHub

## 专业技能
分类：内容

## 工作经验
### 公司名 | 职位 | 起止时间
- 要点一
- 要点二

## 项目经历
### 项目名 | 角色 | 起止时间
一句话项目背景
- 核心工作要点

## 教育背景
### 学校 | 专业 / 学历 | 起止时间
```

格式硬性要求：
- 第 1 行 `# 姓名`；第 2 行职位；第 3 行联系方式用 ` | ` 分隔（缺的字段省略，且只能用真实信息）。
- 板块标题一律用 `## `。
- 每段经历首行 `### 标题 | 角色 | 日期`，用 ` | ` 分三段，日期会自动右对齐。
- 要点用 `- ` 开头，每条一行。
